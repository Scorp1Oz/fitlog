# fitlog — Конспект сесії 4: Акаунти (реєстрація + логін)

> Навчальний підсумок кроку 4: профілі в SQLite, хешування пароля, вхід/реєстрація.
> У кінці — **питання для самоперевірки** і **терміни**.

---

## 0. Що зробили (одним абзацом)

Додали **акаунти**: можна зареєструватися (ім'я + пароль) і ввійти. Профілі
зберігаються в таблиці `profiles` (SQLite). Пароль ніколи не лежить у відкритому
вигляді — лише **сіль** + **SHA-256(сіль+пароль)**. Який профіль активний,
пам'ятаємо між запусками. З'явився новий шар **Auth** («хто залогінений»), що
стоїть *над* замком-біометрією.

---

## 1. Хешування пароля (`src/auth/password.ts`)

Зберігати пароль як є — небезпечно. Замість цього:

```
сіль   = 16 випадкових байтів        (Crypto.getRandomBytesAsync)
хеш    = SHA-256(сіль + пароль)       (Crypto.digestStringAsync)
```

У БД лягають **сіль** і **хеш**. При вході беремо введений пароль, хешуємо з тією
ж сіллю і порівнюємо хеші. Збігаються → пароль вірний.

**Навіщо сіль?** Якби солі не було, однакові паролі давали б однаковий хеш —
і їх можна було б зламати готовими таблицями («rainbow tables»). Сіль робить
кожен хеш унікальним.

> ⚠️ Чесно: SHA-256 — швидкий алгоритм. Для серверного захисту беруть повільні
> (bcrypt/argon2). Поки все локально — цього достатньо; з Firebase цим займеться
> сервер.

> 🔑 Терміни: **hash**, **salt**, **rainbow table**, **SHA-256**.

---

## 2. Репозиторій профілів (`src/db/profiles.ts`)

«Репозиторій» = модуль, що інкапсулює всі SQL-запити до однієї таблиці. Решта
коду не пише SQL — лише викликає функції:

```ts
createProfile(username, password)   // хешує + INSERT
verifyCredentials(username, password) // SELECT + порівняння хешів → Profile | null
getProfileById(id)                   // відновити активний профіль при старті
usernameExists(username)             // дружнє повідомлення при реєстрації
```

Параметри передаємо через `?`-плейсхолдери (`runAsync(sql, value)`) — це захист
від **SQL-ін'єкцій** (значення не вклеюється в текст запиту).

> 🔑 Терміни: **repository pattern**, **parameterized query / SQL injection**,
> `lastInsertRowId`.

---

## 3. Активний профіль (`src/auth/active-profile.ts`)

Щоб після перезапуску не логінитись знову, зберігаємо `active_profile_id` у
SecureStore (у браузері — localStorage, та сама схема, що для замка). При старті
читаємо id → вантажимо профіль із БД.

---

## 4. AuthContext (`src/auth/AuthContext.tsx`)

Спільний стан «хто залогінений» + дії. Стан-машина:

```
loading   → відновлюємо активний профіль при старті
signedOut → ніхто не ввійшов (показуємо екран входу)
signedIn  → профіль активний (показуємо додаток)
```

Дії повертають акуратний результат для UI:
```ts
type AuthResult = { ok: true } | { ok: false; error: string };
```
`register` / `login` при успіху самі ставлять активний профіль і `status`.
`logout` чистить активний профіль → назад на `signedOut`.

> 💡 Той самий патерн, що й `LockContext`: Provider + хук `useAuth()`.

---

## 5. Як шари складаються разом (`_layout.tsx`)

```
DbProvider            ← БД готова
 └ AuthProvider       ← хто залогінений
    └ LockProvider    ← стан біометрії
       └ AuthGate     ← немає профілю?  → екран входу
          └ LockGate  ← увімкнено замок? → екран-замок
             └ Stack   ← сам додаток
```

Порядок не випадковий: **спершу визначаємо профіль, потім замок**. Кожен Provider
стоїть вище за свій Gate (бо Gate читає його через хук).

> 🔎 Нюанс на потім: якщо замок увімкнено, одразу після логіну може зʼявитись і
> екран-замок (подвійний бар'єр). Згодом зробимо, щоб успішний пароль автоматично
> знімав замок.

## 6. Як перевірити

**Веб або Android:**
1. Перший запуск → екран **ВХІД**. Натисни «Немає акаунта? Зареєструватись».
2. Введи ім'я (≥3 символи) і пароль (≥4) → «СТВОРИТИ АКАУНТ» → потрапляєш на home
   з «Привіт, <ім'я>!».
3. «ВИЙТИ З АКАУНТА» → знову екран входу.
4. Увійди тими ж даними → впускає. Невірний пароль → «Невірне ім'я або пароль».
5. Перезапусти додаток (не виходячи) → одразу home (профіль відновлено).

## 8. ✅ Питання для самоперевірки

1. Чому пароль не зберігають у відкритому вигляді? Що зберігаємо натомість?
2. Навіщо потрібна сіль, якщо є хеш?
3. Чому SHA-256 «гірший» за bcrypt для паролів, але нам поки підходить?
4. Що таке `?`-плейсхолдери в SQL і від чого вони захищають?
5. Навіщо окремий «репозиторій» замість SQL прямо в компонентах?
6. Чому шар Auth стоїть вище за Lock, а не навпаки?
7. Як додаток «памʼятає» залогінений профіль після перезапуску?
8. Що поверне `verifyCredentials` при невірному паролі і як це показує UI?
