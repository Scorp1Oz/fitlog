# fitlog — Конспект сесії 3: SQLite + міграційний раннер

> Навчальний підсумок кроку 3: локальна база даних і система міграцій.
> У кінці — **питання для самоперевірки** і **терміни для розбору**.

---

## 0. Що зробили (одним абзацом)

Підключили **SQLite** — локальну базу даних на телефоні, яка стане **єдиним
джерелом правди** для всіх даних fitlog. Написали **міграційний раннер**: він
дивиться, до якої версії схеми вже «докотилась» база, і докочує нові зміни при
оновленні додатку — не псуючи старі дані. Створили **схему v1** з таблицями
профілів, вправ, рутин, сесій і підходів. Профілі закладені одразу, тож кожен
набір даних належить конкретному користувачу.

---

## 1. Навіщо база даних (а не просто файли/змінні)

- **Змінні в пам'яті** зникають при закритті додатку.
- **SecureStore** (з кроку 2) — лише для крихітних секретів (прапорці, паролі).
- **SQLite** — справжня база: таблиці, зв'язки, запити, сортування, тисячі
  рядків. Працює **повністю офлайн**, файл лежить на телефоні.

> 🔑 Термін: **single source of truth** — усе важливе живе в одному надійному
> місці (тут — SQLite), а не розкидане по різних сховищах.

## 2. Сучасний API expo-sqlite

```ts
const db = await SQLite.openDatabaseAsync("fitlog.db");

await db.execAsync(`CREATE TABLE ...`);          // кілька запитів, без відповіді
await db.runAsync("INSERT INTO t (x) VALUES (?)", 5);  // запис; → lastInsertRowId, changes
const row  = await db.getFirstAsync("SELECT ...");      // один рядок
const rows = await db.getAllAsync("SELECT ...");        // масив рядків
await db.withTransactionAsync(async () => { ... });     // транзакція (все або нічого)
```

> 🔑 Термін: **параметри `?`** — значення підставляють окремо (`runAsync(sql, x)`),
> а не «склеюють» у рядок. Це захист від **SQL-ін'єкцій**.

## 3. Що таке міграції і `user_version`

**Проблема:** сьогодні схема одна, завтра ти додаси поле `rpe`. Як оновити базу
в людей, які вже користуються додатком, **не стерши їхні дані**?

**Рішення:** версіонування схеми. SQLite має вбудоване в файл ціле число —
`PRAGMA user_version`. Новий файл = 0. Ми зберігаємо там «до якої версії
докотились».

```ts
// прочитати
const { user_version } = await db.getFirstAsync("PRAGMA user_version");
// записати
await db.execAsync("PRAGMA user_version = 1");
```

**Раннер** ([src/db/runner.ts](../src/db/runner.ts)):
1. читає поточну версію;
2. бере міграції, чия `version > поточної`;
3. кожну виконує **в транзакції** і ставить нову `user_version`.

```
Новий телефон:    version 0 → [миграція v1] → version 1
Наступний запуск: version 1 → немає нових → нічого не робимо (ідемпотентність)
Оновлення в майб: version 1 → [міграція v2] → version 2
```

**Залізне правило:** міграції лише **додають** (нові таблиці/стовпці). Ніколи не
видаляють і не ламають — інакше втратиш чужі дані при оновленні.

> 🔑 Терміни: **migration**, **PRAGMA user_version**, **transaction (atomicity)**,
> **idempotent** (повторний запуск нічого не змінює).

## 4. Схема v1 (таблиці)

| Таблиця | Що зберігає | Ключове |
|---|---|---|
| `profiles` | акаунти користувачів | `username`, `password_hash`, `password_salt` |
| `exercises` | бібліотека вправ | `profile_id` NULL = спільна, інакше — чия кастомна |
| `routines` | програми тренувань | `profile_id NOT NULL` |
| `routine_exercises` | вправи в програмі | `position`, `target_sets`, `rep_low/high` |
| `sessions` | проведені тренування | `profile_id`, `started_at`, `ended_at` |
| `session_sets` | окремі підходи | `weight`, `reps`, `type`, `completed` |

**Чому профілі від початку?** Бо це впливає на ВСІ таблиці: дані прив'язані до
`profile_id`. Якби додали профілі пізніше — довелося б робити складну міграцію й
переписувати наявні рядки. Закласти одразу — дешевше.

> 🔑 Терміни: **foreign key (`profile_id`)**, **index** (пришвидшує пошук за
> `profile_id`), **schema design**.

## 5. Як це під'єднано до додатку

- [src/db/database.ts](../src/db/database.ts) — `initDatabase()`: відкриває базу
  (один екземпляр), вмикає `WAL` + `foreign_keys`, запускає раннер.
- [src/db/DbProvider.tsx](../src/db/DbProvider.tsx) — готує базу при старті; поки
  не готово — чорний фон. Дає її через хук `useDatabase()`.
- [src/app/_layout.tsx](../src/app/_layout.tsx) — `<DbProvider>` стоїть **першим**,
  щоб база була готова раніше за все інше (замок, екрани).

Порядок обгорток: `DbProvider → LockProvider → LockGate → Stack`.

> 🔑 Терміни: **singleton**, **WAL (write-ahead logging)**, **provider pattern**.

---

## 6. Як перевірити

Під час запуску в **терміналі Metro** (де `npx expo start`) має з'явитись:
```
[db] ready, version = 1, tables: exercises, profiles, routine_exercises, routines, session_sets, sessions
```
Це означає: база створилась, міграція v1 пройшла, усі 6 таблиць на місці.
Другий запуск — той самий лог, але міграція вже не виконується (version = 1).

## 8. ✅ Питання для самоперевірки

1. Чому SQLite, а не зберігати все у SecureStore чи у звичайних змінних?
2. Що таке `PRAGMA user_version` і де воно фізично зберігається?
3. Опиши крок за кроком, що робить раннер на новому телефоні (version 0).
4. Чому кожну міграцію виконуємо в транзакції?
5. Що означає «ідемпотентний» і чому повторний запуск раннера безпечний?
6. Чому міграціям можна лише додавати, а не видаляти/змінювати?
7. Навіщо `profile_id` майже в кожній таблиці? Що було б, якби додати профілі пізніше?
8. Навіщо параметри `?` у запитах замість склеювання рядка?
